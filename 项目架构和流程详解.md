## 1. 项目概述

### 1.1 项目简介

**ceph-exporter** 是一个基于 Go 语言开发的 Ceph 集群 Prometheus 指标导出器。

**核心功能**:
- 从 Ceph 集群采集监控指标（容量、性能、健康状态等）
- 将指标转换为 Prometheus 格式并通过 HTTP 接口暴露
- 提供完整的可观测性支持（日志、追踪、指标）

**技术栈**:
- **语言**: Go 1.21+
- **Ceph 交互**: go-ceph 库（CGO 调用 librados）
- **指标导出**: Prometheus client_golang
- **日志系统**: logrus + lumberjack（日志轮转）
- **分布式追踪**: OpenTelemetry + Jaeger
- **日志聚合**: ELK Stack（Elasticsearch + Logstash + Kibana）
- **部署方式**: Docker Compose

### 1.2 项目特点

1. **完整的可观测性**
   - **指标（Metrics）**: 50+ Prometheus 指标，涵盖集群、存储池、OSD、Monitor 等
   - **日志（Logging）**: 结构化日志，支持文件轮转和 ELK 集成
   - **追踪（Tracing）**: OpenTelemetry 分布式追踪，与 Jaeger 集成

2. **高性能设计**
   - CGO 直接调用 Ceph C 库，性能优于命令行方式
   - 并发采集，支持超时控制
   - 异步日志推送，不阻塞主流程

3. **可扩展架构**
   - 插件系统，支持第三方存储监控
   - 模块化设计，易于添加新的采集器

4. **生产就绪**
   - 完整的错误处理和重连机制
   - 优雅关闭，确保资源正确释放
   - 健康检查和就绪检查接口

### 2.2 架构说明

#### 2.2.1 分层架构

ceph-exporter 采用经典的分层架构，从上到下分为：

1. **HTTP 服务层（HTTP Server Layer）**
   - **职责**: 对外提供 HTTP 接口，处理客户端请求
   - **组件**: `internal/server`
   - **接口**:
     - `/metrics`: Prometheus 指标端点，返回所有采集到的指标
     - `/health`: 健康检查端点，用于 Kubernetes liveness probe
     - `/ready`: 就绪检查端点，用于 Kubernetes readiness probe
     - `/`: 首页，显示可用端点列表

2. **采集器层（Collector Layer）**
   - **职责**: 实现 Prometheus Collector 接口，采集 Ceph 指标
   - **组件**: `internal/collector`
   - **采集器列表**:
     - **ClusterCollector**: 集群整体状态（容量、IOPS、PG 统计）
     - **PoolCollector**: 存储池指标（每个池的容量、对象数、IO 速率）
     - **OSDCollector**: OSD 指标（容量、利用率、延迟）
     - **MonitorCollector**: Monitor 指标（存储大小、时钟偏移、仲裁状态）
     - **HealthCollector**: 健康状态指标（集群健康状态、检查项详情）
     - **MDSCollector**: MDS 指标（元数据服务器状态）
     - **RGWCollector**: RGW 指标（对象网关状态）

3. **Ceph 客户端层（Ceph Client Layer）**
   - **职责**: 封装与 Ceph 集群的交互，提供统一的数据获取接口
   - **组件**: `internal/ceph`
   - **功能**:
     - RADOS 连接管理（建立、维护、重连）
     - 执行 Ceph Monitor 命令（如 `ceph status`, `ceph osd df`）
     - JSON 响应解析
     - 错误处理和超时控制

4. **CGO 层（go-ceph）**
   - **职责**: 通过 CGO 调用 Ceph C 库（librados）
   - **组件**: `github.com/ceph/go-ceph/rados`
   - **优势**: 性能优于命令行方式，直接使用 RADOS 协议

5. **支持系统（Supporting Systems）**
   - **配置系统（Config System）**: YAML 配置解析、环境变量覆盖、配置验证
   - **日志系统（Logger System）**: 结构化日志、日志轮转、ELK 集成
   - **追踪系统（Tracer System）**: OpenTelemetry 追踪、Jaeger 集成
   - **插件系统（Plugin Manager）**: 插件加载、生命周期管理、健康检查

#### 2.2.2 数据流向

**指标采集流程**:
```
Prometheus 抓取请求
    ↓
HTTP Server (/metrics)
    ↓
Prometheus Registry (调用所有 Collector)
    ↓
各个 Collector (并发采集)
    ↓
Ceph Client (执行 Ceph 命令)
    ↓
go-ceph (CGO 调用)
    ↓
Ceph Cluster (返回 JSON 数据)
    ↓
Ceph Client (解析 JSON)
    ↓
Collector (转换为 Prometheus Metric)
    ↓
HTTP Server (返回 Prometheus 格式文本)
    ↓
Prometheus (存储指标)
```

**日志流向**:
```
应用代码 (log.Info(...))
    ↓
logrus Logger
    ├─→ 标准输出/文件 (本地日志)
    └─→ Logstash Hook (ELK 集成)
            ↓
        Logstash (TCP/UDP)
            ↓
        Elasticsearch (存储)
            ↓
        Kibana (可视化)
```

**追踪流向**:
```
HTTP 请求
    ↓
Tracing Middleware (创建 Span)
    ↓
业务逻辑 (传递 Context)
    ↓
Tracer Provider (批量导出)
    ↓
OTLP HTTP Exporter
    ↓
Jaeger Collector
    ↓
Jaeger Storage
    ↓
Jaeger UI (可视化)
```

**关键代码位置**:
- 服务器创建: `internal/server/server.go:NewServer()`
- 路由注册: `internal/server/server.go:Start()`
- 追踪中间件: `internal/server/server.go:tracingMiddleware()`
- 健康检查: `internal/server/server.go:healthHandler()`
- 就绪检查: `internal/server/server.go:readyHandler()`

**设计要点**:
1. **中间件模式**: 使用追踪中间件包装所有 Handler，实现横切关注点
2. **ResponseWriter 包装**: 自定义 ResponseWriter 捕获状态码，用于追踪和日志
3. **优雅关闭**: 实现 Shutdown() 方法，确保正在处理的请求完成后再关闭
4. **超时控制**: 配置 ReadTimeout 和 WriteTimeout，防止慢客户端攻击

**设计要点**:
1. **接口实现**: 所有 Collector 实现 `prometheus.Collector` 接口
2. **指标描述符**: 在构造函数中创建 `prometheus.Desc`，描述指标的名称、帮助信息和标签
3. **超时控制**: 每次采集都使用带超时的 Context，防止阻塞
4. **错误隔离**: 单个 Collector 失败不影响其他 Collector
5. **标签使用**: 使用标签区分不同的 OSD/Pool/Monitor（如 `osd="osd.0"`）

### 3.3 Ceph Client 架构

```
┌─────────────────────────────────────────────────────────┐
│            Ceph Client 详细架构                          │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                 Client 结构                              │
│                                                          │
│  • conn: radosConn          (RADOS 连接接口)            │
│  • config: *config.CephConfig (Ceph 配置)               │
│  • log: *logger.Logger      (日志记录器)                │
│  • mu: sync.RWMutex         (读写锁，保护并发访问)      │
│  • closed: bool             (连接关闭标志)              │
└─────────────────────────────────────────────────────────┘
                           │
                           │
        ┌──────────────────┼──────────────────┐
        │                  │                  │
┌───────▼────────┐ ┌───────▼────────┐ ┌──────▼───────┐
│  生命周期管理  │ │  命令执行      │ │  数据获取    │
│                │ │                │ │              │
│ • NewClient()  │ │ • Execute      │ │ • GetCluster │
│ • Connect()    │ │   Command()    │ │   Status()   │
│ • Close()      │ │                │ │ • GetPool    │
│ • Reconnect()  │ │ 执行 Monitor   │ │   Stats()    │
│ • IsConnected()│ │ 命令并返回     │ │ • GetOSD     │
│                │ │ JSON 结果      │ │   Stats()    │
│ 管理 RADOS     │ │                │ │ • GetMonitor │
│ 连接的建立、   │ │ 支持超时控制   │ │   Stats()    │
│ 维护和关闭     │ │ 和错误处理     │ │ • ...        │
└────────────────┘ └────────────────┘ └──────────────┘

┌─────────────────────────────────────────────────────────┐
│              Connect() 连接流程                          │
└─────────────────────────────────────────────────────────┘

1. 创建 RADOS 连接对象
   ↓
   conn, err := newRadosConn(cluster, user)
   • 调用 go-ceph 库创建连接
   • 传入集群名和用户名

2. 读取 Ceph 配置文件
   ↓
   conn.ReadConfigFile(configFile)
   • 通常是 /etc/ceph/ceph.conf
   • 包含 Monitor 地址、认证配置等

3. 设置 Keyring (如果配置了)
   ↓
   conn.SetConfigOption("keyring", keyring)
   • Keyring 包含认证密钥
   • 用于 Cephx 认证

4. 建立连接
   ↓
   conn.Connect()
   • 连接到 Ceph Monitor
   • 完成认证握手

5. 保存连接并标记为已连接
   ↓
   c.conn = conn
   c.closed = false

┌─────────────────────────────────────────────────────────┐
│          ExecuteCommand() 命令执行流程                   │
└─────────────────────────────────────────────────────────┘

1. 检查连接状态
   ↓
   if c.conn == nil || c.closed {
       return error
   }

2. 创建带超时的上下文
   ↓
   ctx, cancel := context.WithTimeout(ctx, timeout)
   defer cancel()
   • 默认超时: 10 秒

3. 在 goroutine 中执行命令
   ↓
   go func() {
       buf, _, err := c.conn.MonCommand(command)
       ch <- result{data: buf, err: err}
   }()
   • 异步执行，避免阻塞
   • MonCommand 执行 Monitor 命令

4. 等待结果或超时
   ↓
   select {
   case <-ctx.Done():
       return timeout error
   case res := <-ch:
       return res.data, res.err
   }

┌─────────────────────────────────────────────────────────┐
│          GetOSDStats() 数据获取示例                      │
└─────────────────────────────────────────────────────────┘

1. 构建 JSON 命令
   ↓
   cmd := {"prefix": "osd df", "format": "json"}
   • prefix: 命令名称
   • format: 返回格式 (json)

2. 执行命令
   ↓
   data, err := c.ExecuteCommand(ctx, cmd)
   • 调用 ExecuteCommand 执行
   • 返回 JSON 字节数组

3. 解析 JSON 响应
   ↓
   var osdDF struct {
       Nodes []OSDStats `json:"nodes"`
   }
   json.Unmarshal(data, &osdDF)
   • 将 JSON 解析为 Go 结构体
   • OSDStats 包含 OSD 的所有统计信息

4. 返回结果
   ↓
   return osdDF.Nodes, nil
```

**关键代码位置**:
- Client 定义: `internal/ceph/client.go`
- RADOS 连接: `internal/ceph/conn_cgo.go`
- 数据结构: `internal/ceph/client.go` (ClusterStatus, OSDStats 等)

**设计要点**:
1. **接口抽象**: 使用 `radosConn` 接口而不是具体类型，便于测试和模拟
2. **并发安全**: 使用 `sync.RWMutex` 保护连接状态
3. **超时控制**: 所有命令执行都支持超时，防止长时间阻塞
4. **错误处理**: 详细的错误信息，包含失败的命令和原因
5. **重连机制**: 提供 `Reconnect()` 方法，支持连接恢复

                └─→ lumberjack.Logger.Write(formatted)
                       │
                       ├─→ 检查文件大小
                       │      ├─ 小于 MaxSize: 直接写入
                       │      └─ 大于等于 MaxSize: 触发轮转

                       │             ├─→ 重命名为 xxx.log.1
                       │             ├─→ 压缩旧文件 (如果启用)
                       │             ├─→ 删除过期文件 (超过 MaxAge)

### 5.2 追踪系统流程

```
┌─────────────────────────────────────────────────────────┐
│            分布式追踪详细流程                             │
└─────────────────────────────────────────────────────────┘

HTTP 请求到达
  │
  ▼
Tracing Middleware
  │
  ├─→ 1. 创建 Span
  │      │
  │      ├─→ tracer.StartSpan(ctx, "GET /metrics")
  │      │      │
  │      │      ├─→ 从 Context 中提取父 Span (如果有)
  │      │      ├─→ 生成 TraceID (如果是根 Span)
  │      │      ├─→ 生成 SpanID
  │      │      ├─→ 记录开始时间
  │      │      └─→ 返回新的 Context 和 Span
  │      │
  │      └─→ defer span.End() (确保 Span 结束)
  │
  ├─→ 2. 设置 Span 属性
  │      │
  │      └─→ tracer.SetAttributes(ctx,
  │             tracer.StringAttr("http.method", "GET"),
  │             tracer.StringAttr("http.url", "/metrics"),
  │             tracer.StringAttr("http.host", "localhost:9128"),
  │             tracer.StringAttr("http.user_agent", "Prometheus/2.x")
  │          )
  │
  ├─→ 3. 注入 Context 到请求
  │      │
  │      └─→ r = r.WithContext(ctx)
  │             (后续处理器可以从 Context 中获取 Span)
  │
  ├─→ 4. 记录 TraceID 到日志
  │      │
  │      ├─→ traceID := tracer.GetTraceID(ctx)
  │      └─→ log.WithTraceID(traceID).Debug("处理请求")
  │             (日志和追踪数据关联)
  │
  ├─→ 5. 包装 ResponseWriter
  │      │
  │      └─→ rw := newResponseWriter(w)
  │             (捕获状态码)
  │
  ├─→ 6. 调用下一个 Handler
  │      │
  │      └─→ next.ServeHTTP(rw, r)
  │             │
  │             ├─→ promhttp.HandlerFor
  │             │      │
  │             │      ├─→ Registry.Gather()
  │             │      │      │
  │             │      │      └─→ Collector.Collect()
  │             │      │             │
  │             │      │             ├─→ 可以创建子 Span
  │             │      │             │      ctx, span := tracer.StartSpan(ctx, "GetOSDStats")
  │             │      │             │      defer span.End()
  │             │      │             │
  │             │      │             └─→ 执行采集逻辑
  │             │      │
  │             │      └─→ 返回 Prometheus 文本
  │             │
  │             └─→ 返回响应
  │
  ├─→ 7. 记录响应状态码
  │      │
  │      └─→ tracer.SetAttributes(ctx,
  │             tracer.IntAttr("http.status_code", rw.statusCode)
  │          )
  │
  ├─→ 8. 设置 Span 状态
  │      │
  │      ├─→ if statusCode >= 500:
  │      │      tracer.SetSpanStatus(ctx, tracer.StatusError, "Server error")
  │      │
  │      ├─→ else if statusCode >= 400:
  │      │      tracer.SetSpanStatus(ctx, tracer.StatusError, "Client error")
  │      │
  │      └─→ else:
  │             tracer.SetSpanStatus(ctx, tracer.StatusOK, "")
  │
  └─→ 9. 结束 Span
         │
         └─→ span.End()
                │
                ├─→ 记录结束时间
                ├─→ 计算持续时间
                └─→ 将 Span 数据发送到导出器
                       │
                       ▼
                TracerProvider (批量导出)
                       │
                       ├─→ 收集多个 Span
                       ├─→ 达到批量大小或超时
                       └─→ 批量导出
                              │
                              ▼
                       OTLP HTTP Exporter
                              │
                              ├─→ 序列化为 OTLP 格式
                              ├─→ HTTP POST 到 Jaeger
                              │      URL: http://jaeger:4318/v1/traces
                              │
                              └─→ 返回结果
                                     │
                                     ▼
                              Jaeger Collector
                                     │
                                     ├─→ 验证数据
                                     ├─→ 存储到后端 (Elasticsearch/Cassandra)
                                     └─→ 可通过 Jaeger UI 查询
```

## 8. 关键技术点

### 8.1 CGO 与 go-ceph

**为什么使用 CGO**:
- **性能**: 直接调用 Ceph C 库（librados），性能优于命令行方式
- **功能**: 可以使用 RADOS 的所有功能，不受命令行限制
- **稳定性**: 使用官方支持的 API，兼容性好

**CGO 的挑战**:
- **编译复杂**: 需要安装 Ceph 开发库（librados-dev）
- **交叉编译困难**: CGO 代码难以交叉编译
- **调试困难**: C 代码的错误可能导致 Go 程序崩溃

**解决方案**:
- 使用 Docker 构建，确保编译环境一致
- 使用接口抽象（`radosConn`），便于测试和模拟
- 详细的错误处理和日志记录

**关键代码**:
```go
// internal/ceph/conn_cgo.go
//go:build cgo
// +build cgo

func newRadosConn(cluster, user string) (radosConn, error) {
    conn, err := cephRados.NewConn()
    if err != nil {
        return nil, err
    }
    return conn, nil
}
```

### 8.2 Prometheus Collector 接口

**Collector 接口**:
```go
type Collector interface {
    // Describe 向 Prometheus 注册指标描述符
    Describe(ch chan<- *Desc)

    // Collect 执行实际的指标采集
    Collect(ch chan<- Metric)
}
```

**实现要点**:
1. **指标描述符**: 在构造函数中创建 `prometheus.Desc`
2. **并发安全**: Collect() 可能被并发调用，需要注意线程安全
3. **错误处理**: 采集失败不应 panic，记录日志即可
4. **超时控制**: 使用 Context 控制采集超时

**示例**:
```go
// internal/collector/osd.go
func (c *OSDCollector) Collect(ch chan<- prometheus.Metric) {
    // 创建超时上下文
    ctx, cancel := newCollectContext()
    defer cancel()

    // 获取数据
    osds, err := c.client.GetOSDStats(ctx)
    if err != nil {
        c.log.Errorf("获取 OSD 统计失败: %v", err)
        return // 不 panic，确保其他 Collector 正常工作
    }

    // 生成指标
    for _, osd := range osds {
        ch <- prometheus.MustNewConstMetric(
            c.up, prometheus.GaugeValue,
            float64(osd.Up()), osd.Name)
    }
}
```

### 8.4 OpenTelemetry 追踪

**追踪概念**:
- **Trace**: 一次完整的请求链路
- **Span**: 链路中的一个操作
- **TraceID**: 追踪的唯一标识符
- **SpanID**: Span 的唯一标识符

**使用示例**:
```go
// 创建 Span
ctx, span := tracer.StartSpan(ctx, "GetOSDStats")
defer span.End()

// 设置属性
tracer.SetAttributes(ctx,
    tracer.StringAttr("osd", "osd.0"),
    tracer.IntAttr("count", 10),
)

// 设置状态
if err != nil {
    tracer.SetSpanStatus(ctx, tracer.StatusError, err.Error())
} else {
    tracer.SetSpanStatus(ctx, tracer.StatusOK, "")
}
```

**追踪与日志关联**:
```go
// 从 Context 中提取 TraceID
traceID := tracer.GetTraceID(ctx)

// 记录到日志
log.WithTraceID(traceID).Info("处理请求")
```

### 8.5 优雅关闭

**优雅关闭流程**:
1. 监听退出信号（SIGINT, SIGTERM）
2. 停止接受新连接
3. 等待现有请求完成（带超时）
4. 关闭所有资源（连接、文件、goroutine）

**实现示例**:
```go
// cmd/ceph-exporter/main.go
// 监听退出信号
quit := make(chan os.Signal, 1)
signal.Notify(quit, syscall.SIGINT, syscall.SIGTERM)

// 阻塞等待信号
sig := <-quit
log.Infof("收到退出信号: %v", sig)

// 创建关闭超时上下文
ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
defer cancel()

// 优雅关闭 HTTP 服务器
if err := srv.Shutdown(ctx); err != nil {
    log.Errorf("HTTP 服务器关闭失败: %v", err)
}

// 关闭其他资源
cephClient.Close()
tp.Shutdown(ctx)
pluginMgr.Close()
log.Close()
```
